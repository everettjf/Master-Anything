# 让「可验证」覆盖任何一个函数 —— 把原始实现当成 oracle

> 这是 [Master-Anything](https://github.com/everettjf/Master-Anything) 的一篇进展介绍。Master-Anything 把任意代码库
> （或文档、PDF）变成一张知识图谱，再用**真实测试**和**图谱事实**来**证明**你已经掌握了它。这一篇讲的是我们怎么补上
> 一个老实的短板：让「可验证的应用（Apply）」不再依赖「恰好有人写过测试」。

![Master-Anything](https://everettjf.github.io/Master-Anything/assets/og.png)

Master-Anything 的整个卖点是一个词：**可验证**。你重新实现一个被挖空的函数，由项目**真实存在的测试套件**判定你是否
通过 —— 不是一个大模型说「答得不错」，而是测试真的变绿。

但说句老实话，这里一直藏着一个短板：**「Apply 可验证」只在恰好有一条测试覆盖那个函数时才成立。** 而现实世界的代码里，
绝大多数函数并没有这种运气 —— 它们没有专属测试。碰到这种函数，旧版本只能退化成「自查」：你写完了，我们也没法客观地告诉你
对不对。对一个号称「可验证」的工具来说，这个洞太大了。

这一版，我们把它补上了。

## 核心思路：别要求有测试 —— 自己造一个 oracle

关键的转念很简单：**当一个函数还没被改动时，它自己就是正确性的事实来源（oracle）。**

于是我们不再去*寻找*一条测试，而是去*生成*一条：

1. 拿**原始实现**，喂给它一批输入，把它的输出当成「黄金答案」记录下来。
2. 用这些「输入 → 黄金答案」生成一条**特征化测试（characterization test）**。
3. 现在把这个函数挖空 —— 那条刚生成的测试立刻就会失败。

就这样，一个原本无法验证的函数，变成了可验证的：你重新实现它，特征化测试通过，你就**客观地**通过了。**任何确定性的、
返回值是「字面量」的函数，都不再需要一条手写测试。**

代码在这里：[`packages/verifier/src/characterize.ts`](https://github.com/everettjf/Master-Anything/blob/main/packages/verifier/src/characterize.ts)。

## 怎么让它「可信」而不是「看起来能跑」

「用原始代码当 oracle」这句话听起来简单，但要让它站得住脚，魔鬼全在细节里。我们加了几道闸门，宁可放弃一个函数，也不生成
一条会骗人的测试：

- **两次运行去抖。** 同一批输入跑两遍，只保留两遍输出**完全一致**的用例。这一步会自动过滤掉读时间、随机数、有副作用的
  函数 —— 它们的输出对不上，就被丢掉，于是绝不会生成一条「时灵时不灵」的测试。
- **`repr` 往返过滤。** 只保留那些返回值能往返通过 `repr`（即能作为一个字面量被断言）的用例。这把范围收敛到返回简单数据
  （数字、字符串、列表、字典……）的函数 —— 也正是「断言它等于某个具体值」最稳的那一类。
- **先证明「挖空会让它失败」。** 在认定一个函数「可验证」之前，我们先把它挖空、把生成的测试跑一遍，确认这条测试**确实**
  抓得住这个空。一条挖了空还能通过的测试是没有意义的，我们直接不采纳。
- **生成的测试是服务端私有的。** 那条特征化测试里嵌着「黄金答案」—— 等于把答案写在了卷子上。所以它绝不下发给前端，API
  返回时被剥掉。你看到的只有挖空后的函数。

## 在产品里长什么样

打开 **Learn → 某个单元 → Apply**，现在这个任务会明确告诉你它是怎么被验证的，一共三种状态：

- **✓ 测试验证** —— 项目里本来就有一条测试覆盖这个函数（一直就支持）。
- **✓ Oracle 验证（自动生成测试）** —— 没有现成测试，于是我们用原始实现当 oracle，**现场合成**了一条特征化测试。**这是
  这次新增的能力。**
- **⚠ 仅自查** —— 这个函数实在没法被安全地特征化（不确定、或者需要复杂入参），于是诚实地标注：通过与否只能你自己判断。

换句话说：以前会落到第三档的一大批函数，现在落到了第二档 —— 真正可验证。

## 范围，以及诚实的边界

可信比吹牛重要，所以把话说清楚。目前覆盖的是：

- **Python、JavaScript、TypeScript** —— 模块级函数，以及**零参构造类**上的方法（比如 `Calculator()` 能直接 new 出来的那种）。
  Python 用 `inspect` + `repr`；JS/TS 用反射 + `JSON` 往返,通过 `node:test` 运行(TS 走 node 内置的类型擦除)。
- 输入来自一组「基本类型 / 集合」的输入电池（battery）；只有**确定性**且**返回字面量**的用例会被保留。

接下来要补的（正按 **A → B → C** 三步走，这是第一步 **A**）：

- **大模型提议输入** —— 接了模型时，让它针对函数的具体语义提一批更有代表性的输入；没接模型时，继续用确定性的输入电池兜底，
  **离线照样能跑**。
- **真实运行轨迹采样** —— 跑仓库自带的示例/入口，在函数边界上采集**真实**的输入输出，而不只是靠模糊测试。

而 **B** 是把现在扁平的「每单元状态机」换成一个会沿着图谱传播证据的**知识追踪**模型，由它来决定「下一题练什么」；**C** 是
**目标驱动的 Quest**：你说出想做的改动（修个 bug、加个功能），系统算出需要掌握的子图，最后用那个真实改动作为终极验证。
完整规划见 [`docs/MASTERY-ROADMAP.md`](https://github.com/everettjf/Master-Anything/blob/main/docs/MASTERY-ROADMAP.md)。
